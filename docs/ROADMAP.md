# Cuprum Roadmap

Дорожная карта ближайших шагов. Текущий статус и план до первой реальной
платы. Видение и обоснование выбора — [`VISION.md`](VISION.md); модель
данных — [`PROJECT.md`](PROJECT.md); канонические термины —
[`GLOSSARY.md`](GLOSSARY.md).

## Где мы сейчас

**Готово:**

- Парс Gerber, растеризация (tiny-skia), композ на экран 15120×6230
- `.goo` энкодер, SDCP клиент (discovery / upload / expose)
- UV-засветка работает в железе (Elegoo Saturn 4 Ultra 16K)
- Tauri UI с превью, multiselect + marquee, align/distribute, persist
- CLI: `discover`, `gerber-info`, `render`, `prepare`, `upload`, `expose`
- Стартовый экран + каталог недавних (SQLite MRU), контейнер `.cuprum` (ZIP)
- Документ проекта: working-dir (распаковка в temp, loose-чтение, recovery/GC),
  автосейв, undo/redo, restore points; `panel` свёрнут в манифест (схема v4)

**Чего нет:**

- Panel-модели в UI: структуры данных (`BoardInstance`/`ToolingHole`, `PanelDoc`
  v2) заведены, но **редактора размещения и Panel-превью ещё нет** (дизайны пока
  не кладутся на панель)
- CNC вообще: ни G-code, ни ToolLibrary, ни auto-leveling
- Ни одной реальной платы в руках

- [x] **Кросс-операционный парс-кеш герберов** (✅ 2026-06-03). На холодном первом показе
  три независимые операции (`board_mesh` / `board_metrics` / SVG-batch) парсили одни и те же
  гербер-байты — конкурентно, т.е. большой медный слой разбирался ~3× одновременно (лишний
  CPU + контеншн за ядра). Добавлен общий `cache::parse_layer_cached`: memory-only кеш
  `Arc<GerberLayer>` по `hash(bytes)` с per-key single-flight — слой парсится один раз на все
  три операции (конкурентные промахи дедупятся, проигравшие ждут и берут готовый `Arc`). Все
  холодные парс-сайты (`svg::render_layer_svg`, `geometry::{layer,region,mask}_polygons`,
  `mesh::edge_segments`, metrics `parse_all`) переведены на него; metrics держит
  `Arc<GerberLayer>` вместо owned. Дисковый тир не нужен (`GerberLayer` не `Serialize`, а
  выходы каждой операции и так персистятся); вывод не меняется → бампы кеш-тегов не нужны.

- [x] **Удешевить prereqs флуша (`compute_valid_keys`)** (✅ 2026-06-04). После оптимизации репака
  (PR #43) `zip_write` перестал доминировать — в `flush` лидировал `compute_valid_keys` (~29 мс на
  плотной плате): `valid_artifact_keys` в `workdir::pack` **перечитывала с диска и хешировала байты
  всех герберов**, а те же байты тут же снова читал `collect_entries` (~13 мс). Два рычага: (1) сам
  cache-хешер (`diskcache::Hasher`) переведён с **MD5 на xxh3-128** — некриптостойкий, но это
  content-ключи артефактов, не граница безопасности; 128 бит держат ту же ширину ключа (32 hex), а
  скорость ~10× (MD5 ~500 МБ/с был корнем стоимости). Ускоряет **все** content-ключи (svg/metrics/
  preview/parse-cache), не только флуш. MD5 остаётся в SDCP-аплоаде (заголовок `S-File-MD5` требует
  именно его). (2) `pack` читает каждый гербер **один раз** (`read_referenced_gerbers`) и
  переиспользует байты и для набора ключей, и для `collect_entries` (карта `rel→bytes`, гербер
  изымается из неё при сборке) — двойного чтения больше нет. Вывод не меняется по смыслу, но смена
  хеша меняет значения ключей → старые артефакты в working-dir/`.cuprum` считаются устаревшими и
  перегенерируются (пользователей пока нет — допустимо). Бамп `METRICS/PREVIEW_VERSION` не нужен
  (версии хешируются как часть ключа, ширина та же).
- **Показывать все хотспоты, а не топ-N.** Сейчас на каждый тип DFM-проблемы
  отдаётся только `HOT_N = 40` находок: `dedup_top` в `geometry.rs` дедупит по
  ~1 мм-ячейке, сортирует worst-first и обрезает (`v.truncate(HOT_N)`). На плотной
  плате реальных нарушений зазора/ширины может быть сотни — пользователь видит
  лишь 40 худших и не может пройтись по всем (навигатор хотспотов в превью
  показывает, напр., «1/40», хотя по факту их больше). Сверловка/via идут отдельным
  списком и режутся иначе — отсюда расхождение в счётчиках. Хочется: видеть **все**
  находки. Развилки на проработку: (а) поднять/убрать `HOT_N` и платить размером
  `BoardMetrics` (едет в `.cuprum` как metrics-артефакт — payload растёт) против
  (б) держать кап в метриках, но грузить полный набор лениво по запросу из UI.
  Любой вариант, меняющий вывод метрик, требует бампа `METRICS_VERSION` в
  `cuprum_core::artifact`. Дедуп по ячейке стоит сохранить (иначе длинный тонкий
  зазор даст десятки почти одинаковых точек) — «все» здесь значит «все после
  дедупа», без обрезания топ-N.
- [x] **Фильтр отображаемых типов проблем на превью** (✅ 2026-06-03). Воронка рядом с DRC-тогглом
  И правый клик по 2D-превью открывают список **актуальных** типов проблем (только те, что реально
  найдены и рисуют хотспоты — зазор/ширина/пережим/поясок/сверло/переходные/маска/шелк) с
  чекбоксами «показывать на превью». Снятая галочка прячет хотспоты этого типа из оверлея И из
  степпера «N», не трогая вердикт. Гранулярность — по «типу проблемы» (мапа `problemTypeOf`: id→
  семейство, top/bottom вместе), не по сырому `category`. Состояние эфемерное (сбрасывается при
  смене дизайна), не персистится. Заведены общие radix-примитивы `Popover`/`Checkbox`/`ContextMenu`
  (первое контекстное меню в приложении — семя паттерна).
