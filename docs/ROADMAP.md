# Cuprum Roadmap

Дорожная карта ближайших шагов. Текущий статус и план до первой реальной
платы. Видение и обоснование выбора — [`VISION.md`](VISION.md); модель
данных — [`PROJECT.md`](PROJECT.md); канонические термины —
[`GLOSSARY.md`](GLOSSARY.md).

## Где мы сейчас

**Готово:**

- Парс Gerber, растеризация (tiny-skia), композ на экран 15120×6230
- `.goo` энкодер, SDCP клиент (discovery / upload / expose)
- UV-засветка работает в железе (Elegoo Saturn 4 Ultra 16K)
- Tauri UI с превью, multiselect + marquee, align/distribute, persist
- CLI: `discover`, `gerber-info`, `render`, `prepare`, `upload`, `expose`
- Стартовый экран + каталог недавних (SQLite MRU), контейнер `.cuprum` (ZIP)
- Документ проекта: working-dir (распаковка в temp, loose-чтение, recovery/GC),
  автосейв, undo/redo, restore points; `panel` свёрнут в манифест (схема v4)

**Чего нет:**

- Panel-модели в UI: структуры данных (`BoardInstance`/`ToolingHole`, `PanelDoc`
  v2) заведены, но **редактора размещения и Panel-превью ещё нет** (дизайны пока
  не кладутся на панель)
- CNC вообще: ни G-code, ни ToolLibrary, ни auto-leveling
- Ни одной реальной платы в руках

Бэклог:
- [x] **Рендер-артефакты как часть проекта** (✅ 2026-06-02): SVG-слои, DFM-метрики и
  цветное превью дизайна стали персистентными content-keyed артефактами в
  `<workdir>/artifacts/{svg,metrics,preview}` — едут внутри `.cuprum` (упаковка
  `workdir::pack`), переданный проект не пересчитывает рендер/метрики (~1с+~6.6с на
  дизайн). Версии артефактов сведены в `cuprum_core::artifact`; устаревшее вычищается
  `artifact::gc` при упаковке. Карточка дизайна рисует одно растровое превью (`<img>`,
  backend-композит через `resvg`) вместо тяжёлого SVG-DOM — фикс торможения фронта на
  большой сетке.
- [x] **Умный ретеншен точек восстановления + GC осиротевших гербер** (✅ 2026-06-02):
  точки несут флаг `auto`; ручные (💾) пиннятся, авто-точки прореживаются по времени
  (GFS: всё <24ч → 1/день·7д → 1/неделю·8нед), устаревшие авто (ссылаются только на
  удалённые дизайны) выбрасываются после 48ч. При `pack` `gc_gerbers` освобождает
  `gerbers/<id>/`, на которые не ссылается ни манифест, ни одна оставшаяся точка —
  переданный проект больше не таскает байты давно удалённых дизайнов. Чистая логика
  (`select_pruned`) тестируема; немигрируемые снимки сохраняются (fail toward
  retention).
- **Удаление дизайна ↔ панель.** Удаление уже снимает ссылку из манифеста, а байты
  осиротевших дизайнов теперь вычищаются GC при упаковке (см. выше). Остаётся: когда
  появится размещение дизайнов на панели (`BoardInstance`), удаление дизайна должно
  снимать и его размещения с панели (и наоборот — нельзя удалить дизайн, «молча»
  оставив висящий placement).
- **Кастомное меню приложения (menu bar).** Своё нативное меню в строке меню
  (macOS) / меню окна (Win/Linux) вместо дефолтного Tauri — **развиваем
  постепенно**. Пункты связываем с действиями `shellStore`: File (Новый проект,
  Открыть, Открыть недавние, Сохранить-точку), Edit (Undo/Redo), View и т.д.
  Tauri 2 `Menu`/`MenuItem` API. Стыкуется с open-by-click и dock-recents
  (Фаза 5) — единая «оболочка» приложения вокруг проекта.
- **Кросс-операционный парс-кеш герберов.** На холодном первом показе дизайна
  три независимые операции (`board_mesh` / `board_metrics` / SVG-batch)
  парсят одни и те же гербер-байты — конкурентно, т.е. большой медный слой
  разбирается ~3× одновременно (лишний CPU + контеншн за ядра). Каждая операция
  уже кеширует свой *результат*, поэтому проблема только холодная. Идея: общий
  кеш `GerberLayer` по `hash(bytes)`, который переиспользуют все три (parse-once
  внутри `board_metrics` уже сделан — это его межоперационное продолжение).
- **Не рендерить SVG-слой по нескольку раз.** Композитное превью дизайна строится
  из тех же per-layer SVG, что и инспектор, но `render_design_preview` рендерит
  слои **заново**, мимо общего SVG-кеша (прямой `svg::render_layer_svg` вместо
  кешированного `cache::layer_svg_artifact`). На холодном показе карточки каждый
  слой рендерится дважды (раз для SVG-артефакта, раз внутри превью). Идея: превью
  переиспользует кешированный per-layer SVG (single-flight дедупит при любом
  порядке) — слой рендерится один раз; маргинальная стоимость превью остаётся
  только компоуз + растеризация. Смежно с парс-кешем выше (там — parse-once, тут —
  render-once).
- **Ускорить самый длинный слой 3D-меша.** После оптимизации сборки меша (PR #38:
  триангуляция грани подложки один раз + `rayon::join` подложки со слоями) потолок
  wall-clock плотной платы ≈ самый длинный поверхностный слой (на `plaid` — Silk
  ~400 мс, медь ~360 мс). Слои уже триангулируются параллельно (`board_geometry`),
  так что общий результат держит **самый дорогой** слой: булевы операции
  `geometry::layer_polygons` / `fill_polygons` (union заливок + разность сверловки).
  Идея: ускорять стоимость одного тяжёлого слоя (профиль фаз внутри `layer_polygons`,
  возможно дешевле union/диффа или меньше тесселяции дуг), а не распараллеливание —
  оно уже есть. Профилировать через mesh-трейсинг (`CUPRUM_TRACE=1`, PR #37).
- **Ускорить сериализацию меша (`pack_board_mesh`).** Упаковка готового меша в
  бинарный wire-формат (positions/normals/indices → LE-байты) на плотной плате
  занимает ~110–180 мс — отдельная фаза вне `board_geometry` (в `main.rs`), сейчас
  не трейсится изнутри и не оптимизирована. Блоб ~15 МБ. Идея: писать байты пачкой
  (`bytemuck`/`extend_from_slice` срезом вместо поэлементного `to_le_bytes`), при
  необходимости — параллельно по секциям.
- **Показывать все хотспоты, а не топ-N.** Сейчас на каждый тип DFM-проблемы
  отдаётся только `HOT_N = 40` находок: `dedup_top` в `geometry.rs` дедупит по
  ~1 мм-ячейке, сортирует worst-first и обрезает (`v.truncate(HOT_N)`). На плотной
  плате реальных нарушений зазора/ширины может быть сотни — пользователь видит
  лишь 40 худших и не может пройтись по всем (навигатор хотспотов в превью
  показывает, напр., «1/40», хотя по факту их больше). Сверловка/via идут отдельным
  списком и режутся иначе — отсюда расхождение в счётчиках. Хочется: видеть **все**
  находки. Развилки на проработку: (а) поднять/убрать `HOT_N` и платить размером
  `BoardMetrics` (едет в `.cuprum` как metrics-артефакт — payload растёт) против
  (б) держать кап в метриках, но грузить полный набор лениво по запросу из UI.
  Любой вариант, меняющий вывод метрик, требует бампа `METRICS_VERSION` в
  `cuprum_core::artifact`. Дедуп по ячейке стоит сохранить (иначе длинный тонкий
  зазор даст десятки почти одинаковых точек) — «все» здесь значит «все после
  дедупа», без обрезания топ-N.
- **Попап выбора отображаемых типов проблем (клик по спиннеру DFM).** Бейдж/спиннер
  DFM на превью (тот, что крутится, пока считаются метрики, и затем показывает
  сводку находок) сделать кликабельным: открывает поповер со списком **актуальных**
  типов проблем (только те категории `Finding`, что реально найдены на этом
  дизайне — зазор, ширина, поясок, сверло, маска, шелк, …) с чекбоксами «показывать
  на превью». Снятая галочка прячет соответствующий слой хотспот-оверлея/подсветки,
  не трогая сам вердикт. Чисто фронтовое (состав находок уже считает
  `feasibility.ts`; нужно состояние видимости по категориям + фильтрация оверлея в
  превью/инспекторе). Лоск, но заметно влияет на восприятие плотных плат — можно
  изолировать одну проблему и не тонуть в общей мешанине точек.

- [ ] **G-code emitter** для GRBL 1.1:

      - M6 как manual pause для смены инструмента

- [ ] **Auto-leveling**:
      - probing UI (зажим-крокодил к меди + щуп)

      - построение HeightMap
      - warp-компенсация Z на этапе эмиссии G-code
- [ ] **Редактор ProcessStep'ов**: массив, drag для переупорядочки, кнопки

## Backlog `[не делать пока]`

Сюда то, что **не** в плане ближайшего года:

- Multilayer (4+ слоя) — дома практически невозможно
- Pattern plating (требует tin-ванну и второй цикл резиста)
- Surface finish (ENIG / HASL / OSP) — промышленный процесс
- DRC / netlist-валидация — это работа KiCad, не Cuprum
- Импорт чего-либо кроме Gerber + Excellon (ODB++, IPC-2581)
- Электрический тест (ICT, flying probe)
